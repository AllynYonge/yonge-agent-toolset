# 提示词工作台 (Prompt Workbench)

## 参数化执行模式

支持通过 `--{mode} {sub-mode}` 参数快捷调用特定功能，跳过模式检测直接进入执行：

```
/prompt-assitor --optimize trim    # 精简优化：保持功能不变，重组结构、消除歧义、规范术语
/prompt-assitor --optimize full    # 完整优化：道·法·术三层深度优化（默认优化模式）
/prompt-assitor --create           # 创建模式
/prompt-assitor --analyze          # 分析模式
```

### 参数解析规则

1. 检测用户输入是否包含 `--` 参数标记
2. 若包含参数 → 直接路由到对应模式，无需额外确认
3. 若不包含参数 → 进入下方「模式检测」逻辑
4. `--optimize` 不带子模式时，默认为 `full`

### `--optimize` 子模式

| 子模式 | 定位 | 参考文档 |
|--------|------|---------|
| `trim` | 保持功能不变，专注结构重组、歧义消除、术语规范 | [trim-optimize.md](references/trim-optimize.md) |
| `full` | 道·法·术三层深度优化，可能重塑核心意图 | [optimize-prompt.md](references/optimize-prompt.md) |

## 模式检测

当用户未使用参数化指令时，通过以下信号判断模式：

| 模式 | 触发信号 | 参考文档 |
|------|----------|---------|
| **创建** | "帮我写/创建/生成一个提示词"、提供需求描述但无已有提示词 | [create-prompt.md](references/create-prompt.md) |
| **优化** | "帮我优化/改进/修改这个提示词"、提供已有提示词 + 改进要求 | [optimize-prompt.md](references/optimize-prompt.md) |
| **分析** | "帮我分析/评估/拆解这个提示词"、要求理解提示词设计 | [analyze-prompt.md](references/analyze-prompt.md) |

无法判断时，询问用户。确定模式后，阅读对应参考文档获取详细方法论。

## 创建模式 (快速指引)

### 工作流

1. **需求分类与深层目标挖掘**
   - 识别五大需求类型：决策 `G+O+C` / 分析 `Q+D+M` / 创作 `T+S+ID` / 验证 `O+VM+RP` / 执行 `T+SC+OF`
   - 分析表面请求背后的底层目标（"马车 vs 汽车"原则——用户要马车，真正需要的可能是更快的交通方式）
   - 评估战略清晰度：目标是否明确？成功标准是否清晰？

2. **信息完备性检查**
   - 按需求类型框架组件逐项核查
   - 战略模糊 → 在 `<补充信息请求>` 中提出诊断性问题
   - 战略清晰但缺少输入数据 → 定义为变量 `{$VAR}` 并继续

3. **提示词构建（规范结构）**
   - 角色设定 → 上下文与输入（XML标签包裹） → 思考步骤（`<thinking>`/`<scratchpad>`） → 约束规则（正向+负向） → 输出格式 → 错误处理
   - 大段输入放在处理指令之前
   - 推理先于输出：先提取依据，再给最终答案

4. **质量验证**（内部执行，不展示给用户）
   - 底层目标对齐度 = 高
   - 客观信息完整性 = 是
   - 约束一致性 = 是

### 输出格式

```xml
<分析>
[需求分类、底层目标、信息完备性]
</分析>

<补充信息请求>  <!-- 仅信息不完整时 -->
[精准问题]
</补充信息请求>

<生成的提示词>  <!-- 信息完整时 -->
[完整可用的提示词]
</生成的提示词>
```

## 优化模式 (快速指引)

### 道·法·术三层框架

| 层级 | 含义 | 关注点 |
|------|------|--------|
| **道** | Why | 核心意图、最终价值、战略目的 |
| **法** | How | 原则、策略、交互模式（CoT/角色扮演/模板输出等） |
| **术** | What | 具体技巧（XML标签/精确动词/否定约束/格式规范等） |

### 工作流

1. **意图分析（道的探索）**：分析原始提示词 + AI回答 + 用户反馈三角关系
2. **问题诊断**：歧义/冗余/冲突/缺失 + 结构/逻辑/清晰度
3. **策略制定**：重塑道 → 选择法 → 设计术（每个术明确关联其服务的法和道）

### 输出格式

```xml
<thinking>
[道法术三层完整分析过程]
</thinking>

<optimized_prompt>
[优化后完整提示词]
</optimized_prompt>
```

## 分析模式 (快速指引)

### 工作流

1. 通读并完全理解目标提示词
2. 按道·法·术每个维度，从原文中寻找证据
3. 对每条证据评价优劣
4. 生成结构化分析报告（严格遵循报告模板）
5. 提炼可复用方法论

### 输出格式

````markdown
# 提示词分析报告

### 1. 总体评价
- **一句话亮点：** [核心优点]
- **一句话短板：** [最大改进空间]

### 2. "道" (战略意图) 分析
- **优点：** [基于原文证据]
- **可改进点：** [基于原文证据]

### 3. "法" (系统架构) 分析
- **优点：** [基于原文证据]
- **可改进点：** [基于原文证据]

### 4. "术" (执行技巧) 分析
- **优点：** [基于原文证据]
- **可改进点：** [基于原文证据]

### 5. 核心洞察与改进建议
- **可复用方法论：** [抽象出的设计模式]
- **具体改进建议：** [明确的修改方案]
````

## 核心质量标准（所有模式共享）

1. **第一性原理**：穿透表面直击底层需求
2. **结构化强制**：清晰模块 + XML标签 + 明确格式
3. **信息密度最大化**：精炼高效，无废话
4. **约束完备**：正向约束 + 负向约束
5. **推理先于输出**：先 `<thinking>` 后 `<answer>`
6. **所有输出为中文**
