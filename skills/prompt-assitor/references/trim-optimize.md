# Trim 优化模式 - 详细方法论

## 核心定位

在**不改变原有提示词功能和意图**的前提下，通过结构重组、歧义消除和规范化处理，提升提示词的清晰度、可维护性和执行可靠性。

**与 full 模式的区别**：
- full 模式可能重塑"道"（核心意图），引入新策略和技巧
- trim 模式严格保持"道"不变，仅在"法"和"术"层面做减法和整理

## 约束边界

| 允许 | 禁止 |
|------|------|
| 调整段落顺序和层级结构 | 增删核心功能或改变意图 |
| 合并重复/冗余指令 | 引入原文不存在的新策略 |
| 将模糊表述替换为精确表述 | 改变输出格式的核心要求 |
| 统一术语和概念命名 | 删除有效的约束条件 |
| 补充缺失的章节分隔 | 添加原文未暗示的新约束 |
| 修正逻辑矛盾和冲突 | 改变角色设定的本质 |

## 工作流

### 步骤1：功能基线提取

逐段阅读原始提示词，提取并记录：

1. **核心意图**：这个提示词要完成什么任务？
2. **功能清单**：列出所有独立功能点（每个功能用一句话描述）
3. **约束清单**：列出所有正向和负向约束
4. **输出规格**：期望的输出格式和结构

此基线作为优化后的验证标准——优化结果必须覆盖基线中的每一项。

### 步骤2：结构诊断

从以下四个维度诊断问题：

#### 2.1 结构问题

- **层级混乱**：同级内容嵌套不当，父子关系不清
- **顺序失当**：依赖关系倒置（后文引用前文未定义的概念）
- **模块边界模糊**：不同职责的内容混杂在同一段落
- **缺少导航标记**：大段文本无标题、无分隔，增加认知负荷

#### 2.2 歧义问题

- **指代不明**："它"、"这个"、"上述"指向不清
- **量化模糊**："适当"、"一些"、"尽量"缺乏具体标准
- **条件不完整**：if 无 else，触发条件边界不清
- **多义词**：同一词在不同位置含义不同

#### 2.3 冗余问题

- **重复指令**：同一要求在多处以不同措辞出现
- **过度解释**：已经清晰的指令附带冗长说明
- **无效修饰**：不影响执行的形容词和副词

#### 2.4 规范问题

- **术语不一致**：同一概念使用多种名称（如"用户/客户/使用者"混用）
- **格式不统一**：列表、标题、标签风格不一致
- **语气混杂**：祈使句、陈述句、疑问句混用于指令中

### 步骤3：执行优化

按以下优先级依次处理（高优先级问题优先）：

1. **修正逻辑矛盾** → 消除相互冲突的指令
2. **统一术语** → 选定每个概念的规范名称，全文替换
3. **消除歧义** → 将模糊表述替换为精确、可验证的描述
4. **重组结构** → 按执行流程或逻辑依赖关系重新排列
5. **合并冗余** → 将重复指令合并为单一权威表述
6. **补充分隔** → 添加标题、XML标签或分隔线明确模块边界
7. **精简措辞** → 在不损失信息的前提下缩减字数

### 步骤4：基线验证

逐项核对步骤1提取的功能基线：

- [ ] 每个功能点在优化后仍然存在且表述清晰
- [ ] 每个约束条件未被删除或弱化
- [ ] 输出格式要求完整保留
- [ ] 核心意图未发生偏移

若任何一项不满足，回退修正。

## 输出规范

```xml
<trim_analysis>
## 功能基线
[提取的核心意图、功能清单、约束清单]

## 诊断发现
- 结构问题：[具体问题列表]
- 歧义问题：[具体问题列表]
- 冗余问题：[具体问题列表]
- 规范问题：[具体问题列表]

## 优化操作摘要
[按优先级列出执行的具体操作，每项标注类型]
</trim_analysis>

<optimized_prompt>
[优化后的完整提示词]
</optimized_prompt>

<verification>
## 基线验证
- 功能覆盖：✓/✗ [逐项确认]
- 约束保留：✓/✗ [逐项确认]
- 意图一致：✓/✗
</verification>
```

## 典型 Trim 操作示例

### 示例1：术语统一

**优化前**：
> 当用户提交请求时...客户发送的内容...使用者输入的信息...

**优化后**：
> 当用户提交请求时...用户发送的内容...用户输入的信息...

（统一使用"用户"指代同一对象）

### 示例2：歧义消除

**优化前**：
> 生成适当长度的回复

**优化后**：
> 生成 200-500 字的回复

（将模糊量化替换为具体范围；若原文无法推断具体值，则标注为需用户确认的变量 `{$REPLY_LENGTH}`）

### 示例3：结构重组

**优化前**：
> 你是一个助手。输出用JSON格式。先分析用户意图。注意不要输出敏感内容。分析完后给出建议。JSON要包含analysis和suggestion字段。

**优化后**：
> **角色**：你是一个助手。
>
> **执行流程**：
> 1. 分析用户意图
> 2. 基于分析给出建议
>
> **输出格式**：JSON，包含 `analysis` 和 `suggestion` 字段。
>
> **约束**：不要输出敏感内容。

（将混杂的指令按职责拆分为独立模块）
