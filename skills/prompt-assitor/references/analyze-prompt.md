# 分析提示词 - 详细方法论（道·法·术分析框架）

## 分析目标

解构、评估和提炼目标提示词中的设计智慧，沉淀可复用的方法论。所有评价必须有原文作为论据，杜绝空话。

## 道·法·术分析维度

### 一、道 (The "Why") - 战略意图与核心价值

| 维度 | 分析要点 |
|------|---------|
| **战略意图** | 服务什么宏观目标（效率/创意/合规等）？产出用于何种场景？ |
| **核心价值** | 最独特的价值点是什么？是否构建了难以模仿的"壁垒"？ |
| **设计哲学** | 偏向精确控制还是开放探索？追求确定性还是涌现性？ |

### 二、法 (The "How") - 系统架构与方法论

| 维度 | 分析要点 |
|------|---------|
| **系统架构** | 结构是否完整、模块化？是否包含：角色设定、背景上下文、核心任务、规则约束、工作流/步骤、示例引导、输出格式化 |
| **逻辑自洽性** | 各模块间组织是否清晰、顺畅、无矛盾？ |
| **可复用方法论** | 架构能否被抽象为通用的"提示词模式"(Prompt Pattern)？ |

### 三、术 (The "What") - 执行技巧与具体措辞

| 维度 | 分析要点 |
|------|---------|
| **精确措辞** | 动词、量词、术语是否精准无歧义？ |
| **分隔符与结构化** | 是否有效使用XML标签、Markdown等降低AI理解成本？ |
| **激励与压力** | 是否使用了能激发AI潜能的措辞？ |
| **内部推理与验证** | 是否要求模型先在内部分析、自检，再输出必要的依据摘要或验证结果？ |
| **示例质量** | 示例是否典型、清晰、覆盖边界情况？ |

## 分析工作流

### 第一步：深度阅读与证据提取

1. 通读并完全理解 `<prompt_to_analyze>` 中的目标提示词
2. 针对道·法·术每一个分析维度，逐条从原文中寻找**直接证据**（引用原文）
3. 对每条证据判定：是优点还是不足？理由是什么？

将思考过程记录在 markdown 代码块中：


## 分析质量要求

1. **证据驱动**：每个评价点必须引用原文作为论据，禁止空泛评价
2. **深度优先**：追求洞察深度而非覆盖广度，抓住本质问题
3. **可操作性**：改进建议必须具体到可直接执行的修改方案
4. **方法论沉淀**：从每个分析中提炼可推广的设计模式
5. **"壁垒"思维**：评估提示词是否创建了独特的、难以复制的价值
