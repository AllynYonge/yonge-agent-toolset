# 创建提示词 - 详细方法论

## 第一阶段：需求分析与战略洞察

### 1.1 五大需求类型分类

收到用户需求后，先判断属于哪种类型，并使用对应框架组件检查信息完备性：

| 需求类型 | 定义 | 框架组件 |
|---------|------|---------|
| **决策** | 权衡利弊、评估风险，选择最优方案 | `目标(G)` + `选项(O)` + `评判标准(C)` |
| **分析** | 深度解读数据/信息，揭示规律、因果 | `问题(Q)` + `数据/信息(D/I)` + `分析方法(M)` |
| **创作** | 生成原创内容（文本、代码、创意等） | `主题(T)` + `风格/约束(S/C)` + `创新方向(ID, 可选)` |
| **验证** | 审查已有结论/方案/代码的正确性、可行性 | `审查对象(O)` + `验证方法(VM)` + `风险点(RP)` |
| **执行** | 严格按指令完成程序化/操作性任务 | `任务(T)` + `步骤/约束(SC)` + `输出格式(OF)` |

### 1.2 深层目标挖掘（"马车 vs 汽车"原则）

分析用户请求背后的底层目标，而非仅响应字面需求：

- 用户直接说的是什么？（表面请求）
- 他们真正想解决的问题是什么？（底层目标）
- 表面请求是否是达成底层目标的最佳路径？

如果存在更优路径，在尊重原始请求的前提下，巧妙地融入更优方案。

### 1.3 战略清晰度评估（三分支决策）

即使所有输入参数都已就绪，核心目标、成功标准、目标受众、关键差异化点、隐含约束是否仍然模糊？

- **战略模糊** → 提出诊断性问题引导用户澄清价值主张、障碍和差异化点。仅在 `<补充信息请求>` 中输出
- **战略清晰但缺少输入数据** → 将缺失内容定义为变量 `{$VARIABLE_NAME}` 并直接进入第二阶段
- **全部清晰** → 直接进入第二阶段

### 1.4 诊断性问题库

当需要深度探查底层需求时，从以下问题中选择最相关的提问：

- **受众探查**：最终受众是谁？希望他们产生什么想法/感受/行动？
- **成功定义**：用一句话描述空前成功的画面是什么？
- **价值探查**：完美完成此任务，将解决的最根本问题或创造的最大价值是什么？
- **边界挑战**：无任何限制时的理想结果？必须做取舍时，什么核心要素绝不能牺牲？
- **动机探查**：是什么促使现在解决这个问题？背后是否有更宏大的目标？

## 第二阶段：提示词生成

### 2.1 定义最小输入变量集

- 确定AI执行任务所需的最小、不重叠的文本输入变量集
- 格式：`{$VARIABLE_NAME}` 或 `{{VARIABLE_NAME}}`
- 预期包含长文本的变量应放在处理指令之前

### 2.2 规范结构（按顺序组织）

生成的提示词必须包含以下模块（按需取舍，但顺序不变）：

1. **角色设定**：定义AI扮演的专家人格
2. **上下文与输入**：用XML标签清晰包裹输入变量（如 `<document>{$DOC}</document>`）
3. **思考步骤**：对复杂任务，要求AI在内部推理区域先行思考
   - 关键原则：先在内部完成分析、规划和自检，再给出最终答案
   - 默认不要输出完整思维链；如需展示过程，只输出 `<analysis_summary>`、`<rationale>` 或 `<verification>` 这类简短摘要
4. **约束与规则**：正向约束（必须做什么）+ 负向约束（不许做什么）
5. **输出格式**：精确指定标签和结构（如 `将答案放在 <answer> 标签内`）
6. **错误处理**：无法完成任务时的行为指引

### 2.3 高级技巧

- **变量参数化**：使提示词可复用，输入通过 `{$VAR}` 注入
- **XML标签结构化**：降低AI理解成本，清晰分隔不同信息区域
- **示例引导**：按任务复杂度提供 1-5 个 `<example>`，展示期望的输入-输出对，覆盖正常、边界、反例和输出格式场景
- **位置优化**：最重要的指令放在开头和末尾
- **能力激励**：措辞旨在激发AI探索其能力边界

### 2.3.1 Few-shot 示例选择规则

仅在示例能显著降低歧义或稳定格式时加入，不为简单任务堆砌样例。

| 示例类型 | 使用场景 | 作用 |
|----------|----------|------|
| **正常样例** | 用户期望固定风格、格式或判断路径 | 展示标准输入到标准输出的映射 |
| **边界样例** | 存在长度、数据缺失、异常输入或灰区判断 | 告诉模型边界条件下如何处理 |
| **反例** | 用户明确不想要某类输出，或任务容易误判 | 标出错误做法和拒绝/修正方式 |
| **格式样例** | 输出需要被程序读取或复制到固定模板 | 锁定字段、顺序、标签和格式细节 |

### 2.4 概念激活增强（可选）

对复杂需求，可先内部生成核心关键词"概念激活集"以增强语义精准度：

- **A类 - 潜在需求**：用户未言明的痛点、焦虑、渴望
- **B类 - 核心概念**：领域基础概念、前沿趋势、方法论术语

**概念升级技巧**：将具体/口语化表达提升为抽象、高激活度术语。将这些概念融入角色设定、上下文和指令中。

## 质量验证（内部检查清单）

| 检查项 | 标准 |
|--------|------|
| 底层目标对齐度 | 必须为"高" |
| 客观信息完整性 | 必须为"是" |
| 约束一致性 | 必须为"是" |
| 结构完整性 | 角色+上下文+指令+约束+格式 |
| 内部推理先于输出 | 复杂任务有内部分析、自检和简短验证摘要 |

## 输出规范

### 信息不完整时

```xml
<分析>
[需求分类、底层目标、信息完备性]
</分析>

<补充信息请求>
[精准问题]
</补充信息请求>
```

### 信息完整时

```xml
<分析>
[需求分类、底层目标、信息完备性]
</分析>

<生成的提示词>
[完整可用的提示词]
</生成的提示词>

<verification>
[简短核对底层目标、输入变量、约束、输出格式和风险边界]
</verification>
```

## 参考示例模式

### 模式A：FAQ客服代理
角色设定 → FAQ作为输入 → 内部提取引用 → `<answer>` 回答 → `<verification>` 核对范围

### 模式B：文档问答
文档输入 → 提取编号引文 → 基于引文回答（末尾标注编号）→ 无法回答时声明

### 模式C：苏格拉底式辅导
角色+内部独白 → 每次回应前内部求解验证学生答案 → 根据对错给提示非答案

### 模式D：函数调用代理
`<scratchpad>` 规划 → `<function_call>` 执行 → `<function_result>` 接收 → 错误重试 → `<answer>` 回答
