# SKILL.md 生成模板

以下是生成单个技能文件时的完整模板。占位符用 `{{}}` 标记，生成时替换为实际内容。

---

## YAML 前置元数据

```yaml
---
name: {{skill-name}}
description: >
  {{一句话职责描述}}。当用户{{触发条件描述}}时，应使用本技能。
  {{补充触发场景，2-3 个具体情况}}。
license: MIT
metadata:
  author: meta-workflow-generator
  version: "1.0"
---
```

**description 写作要求：**
- 第一句说清楚"这个技能做什么"
- 第二句说清楚"什么时候该触发"
- 使用第三人称（"本技能应在...时使用"）
- 具体到可判断，避免"当用户需要帮助时"这种万金油表述

---

## 编写原则

1. **祈使句/不定式** — "To accomplish X, do Y"，不用"你应该..."
2. **具体可操作** — 每个步骤要具体到"做什么"和"怎么判断做完了"
3. **领域方法论** — 核心理念部分必须包含该领域的专业方法论，不是通用废话
4. **控制长度** — SKILL.md 正文 ≤ 3000 字，详细内容放 references/
5. **不写评论** — 不在模板中写"此处可扩展"之类的元注释

---

## references/ 文件约定

| 文件名 | 用途 | 长度建议 |
|---|---|---|
| `output-template.md` | 该技能产出物的格式模板 | 按需，可长 |
| `domain-guide.md` | 领域专业知识和方法论参考 | ≤ 5000 字 |
| `examples.md` | 该技能的输入/输出示例 | 1-2 个完整示例 |
| `checklist.md` | 详细的质量检查清单（SKILL.md 放摘要） | 按需 |

references/ 文件不需要 YAML 前置元数据，直接用 Markdown 标题开始。

## 工作流协同章节模板

每个生成的技能末尾**必须**包含 `## 工作流协同` 章节，声明上下游衔接关系。这是"可选加速路径"——上游产出存在时自动利用，不存在时回退到独立模式。

```markdown
## 工作流协同

### 上游
- **前置技能：** {prefix}{N-1}-{name}
- **输入产出物：** `{路径}/{文件名}`
- **输入格式要求：** {必须包含的字段/章节}
- **缺失时降级：** {如果上游产出不存在，本技能如何处理——提供输入引导或使用默认值}

### 下游
- **后续技能：** {prefix}{N+1}-{name}
- **输出产出物：** `{路径}/{文件名}`
- **输出契约：** {下游技能依赖的关键字段/章节，不可省略}
- **触发建议：** {完成本技能后建议用户执行的下一步}

### 文件路径约定
- 工作目录：由用户指定或使用当前工作目录
- 本技能产出物路径：`{工作目录}/{具体文件名}`
- 命名规则：`{prefix}{N}-{描述性名称}.md`
```

**写作约束：**
- 上游"输入格式要求"写"需要什么"，而非"读取技能 X 的输出"
- "缺失时降级"必须提供具体的降级方案（提问用户 / 使用默认模板 / 跳过非关键字段）
- 首/末节点只写单向（首节点无上游，末节点无下游）

## 交付目录结构

生成的技能文件按以下结构组织：

```
.claude/skills/
├── {prefix}1-{name}/
│   ├── SKILL.md
│   ├── references/
│   │   ├── output-template.md    (产出物格式模板)
│   │   └── domain-guide.md       (领域方法论参考)
│   └── scripts/                   (可选)
├── {prefix}2-{name}/
│   ├── SKILL.md
│   └── references/
├── ...
└── {prefix}-orchestrator/         (仅在满足编排器生成条件时创建)
    ├── SKILL.md
    └── references/
```

**scripts/ 目录：** 当某环节包含可确定性执行的重复操作时（格式转换、数据校验、模板渲染、文件批量处理），将这些操作封装为脚本。脚本的优势是 token 高效、执行确定性强，且可在不加载到上下文的情况下直接执行。仅在确有需要时创建，不强制每个技能都有 scripts/ 目录。

## 脚本增强（可选）

当技能中的某些步骤适合用脚本而非 LLM 完成时，将其封装为 `scripts/` 下的可执行脚本。脚本由 SKILL.md 在特定步骤中通过 Bash 工具调用，不进入上下文窗口，省 token 且执行结果稳定。

### 何时使用脚本

| 场景 | 说明 | 示例 |
|---|---|---|
| 确定性重复操作 | 输入→输出映射固定，无需判断 | 格式转换、模板渲染、schema 校验 |
| LLM 能力边界外 | LLM 无法可靠完成的操作 | 图片处理、二进制文件操作、大文件聚合统计、调用外部 API/CLI 工具 |
| Token 密集型处理 | 数据量大但逻辑简单 | 批量文件重命名、目录结构生成、日志解析聚合 |
| 精确计算 | 需要数值精度保证 | 哈希校验、diff 统计、依赖版本比对 |

| 不适合脚本化 |
|---|
| 需要语义理解的分析判断 |
| 需要上下文推理的内容生成 |
| 需要与用户交互的决策点 |
| 输出依赖 LLM 创造力的环节 |

**判断标准：** 操作是确定性的，或者 LLM 不具备完成该操作的能力 → 用脚本。需要"判断"或"创作" → 留给 LLM。

### 文件结构与命名

```
scripts/
├── validate-output.sh       # 校验产出物是否符合规范
├── generate-scaffold.sh     # 基于模板生成文件骨架
├── transform-format.py      # 格式转换
└── README.md                # 脚本用途索引（一行一个）
```

**命名规则：** `{动词}-{对象}.{ext}`，动词用祈使式（validate、generate、transform、check）。

**语言选择：**
- Shell（bash/zsh）：文件操作、命令编排、简单文本处理
- Python：复杂数据转换、JSON/YAML 处理、需要第三方库时
- Node.js：与项目前端技术栈一致时，或需要处理 JS/TS AST 时

### 脚本编写原则

1. **幂等** — 多次执行同一输入产生相同结果，不累积副作用
2. **参数化** — 通过命令行参数或环境变量接收输入，不硬编码路径
3. **退出码语义** — `0` 成功，`1` 业务错误（校验不通过），`2` 运行时错误（缺少依赖）
4. **输出分离** — 结果写 stdout，诊断信息写 stderr，便于管道组合
5. **零依赖优先** — 尽量只用语言标准库；需要外部依赖时在 README.md 注明安装方式
6. **只做一件事** — 每个脚本对应一个原子操作，复杂流程由 SKILL.md 编排多个脚本

### 在 SKILL.md 中声明脚本调用

在对应步骤的"操作"中明确标注何时调用、传什么参数、如何处理输出：

```markdown
### 步骤 3：校验产出物

**操作：**
1. 执行校验脚本：
   ```bash
   bash scripts/validate-output.sh --input "$OUTPUT_PATH" --schema references/output-schema.json
   ```
2. 退出码为 0 → 进入下一步
3. 退出码为 1 → 读取 stderr 中的校验错误，修正后重试（最多 2 次）
4. 退出码为 2 → 提示用户检查环境依赖
```

### 脚本与探测式输入

脚本也可用于执行初期的环境探测，输出结构化结果供 LLM 决策：

```bash
# scripts/detect-stack.sh — 输出 JSON 格式的技术栈信息
#!/bin/bash
result="{}"
[ -f package.json ] && result=$(echo "$result" | jq '. + {"runtime":"node","pkg":true}')
[ -f pom.xml ] && result=$(echo "$result" | jq '. + {"runtime":"java","build":"maven"}')
[ -f Cargo.toml ] && result=$(echo "$result" | jq '. + {"runtime":"rust","build":"cargo"}')
echo "$result"
```

**探测规则：**
- 探测脚本必须是只读操作，不修改任何文件
- 输出 JSON 或 key=value 格式，便于 LLM 解析
- 探测失败时回退到用户输入，不阻塞执行

## 行为适配章节模板

每个生成的技能应包含行为适配设计（≥ 1 个自适应维度）：

```markdown
## 行为适配

### 模式检测

| 信号 | 检测方式 | 触发模式 |
|---|---|---|
| {信号1} | {如何判断} | → {模式名} |
| {信号2} | {如何判断} | → {模式名} |
| 默认 | — | → {默认模式} |

### {模式A} 与 {模式B} 差异

| 维度 | 模式A | 模式B |
|---|---|---|
| 步骤数 | {N} | {M} |
| 输出详细度 | {级别} | {级别} |
| 用户确认点 | {N} | {M} |
```

**常见自适应维度：**

| 维度 | 适用场景 | 示例 |
|---|---|---|
| 输入复杂度 | 简单 vs 复杂任务 | 轻量模式跳过某些步骤 |
| 技术栈/领域 | 不同技术选择影响流程 | REST vs GraphQL → 不同接口设计流程 |
| 项目阶段 | 新建 vs 迭代 | 新建需完整设计，迭代只做增量 |
| 用户角色 | 新手 vs 专家 | 调整引导详细度和确认频率 |
| 输入来源 | 上游产出 vs 手动输入 | 有上游产出时自动提取，否则引导填写 |
