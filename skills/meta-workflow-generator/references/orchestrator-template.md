# 编排技能生成条件

本模板仅在以下条件**同时满足**时使用：

| 条件 | 说明 |
|---|---|
| 技能节点 ≥ 6 个 | 4-5 个节点通过各技能的"工作流协同"章节即可衔接 |
| 存在非线性依赖 | 有分支、并行、循环中的至少一种；纯线性流水线不需要编排器 |
| 用户需要端到端执行 | 若用户只使用单个技能，编排器无意义 |

不满足以上条件时，**不生成编排技能**，仅依赖各技能的"工作流协同"章节声明衔接关系。

**生成检查要点：**
- 状态机覆盖所有技能节点
- 进度判断逻辑明确（如何确定下一个应执行的技能）
- 上下文注入规则为每个技能定义了具体内容
- 异常处理覆盖了所有 failed/blocked 路径
- 质量门禁配置合理（关键路径严格，辅助节点宽松）
- 知识沉淀规则：技能重试 ≥ 2 次、门禁退回修正、用户修正产出物时自动记录

# 编排技能模板

以下是完整模板，替换 `{{}}` 占位符后使用。

---

## SKILL.md 模板

```yaml
---
name: {{prefix}}-orchestrator
description: >
  {{领域名称}}工作流编排器。负责协调 {{prefix}} 系列技能的执行顺序、
  上下文传递和质量门禁。当用户希望端到端执行完整的{{核心任务描述}}工作流时使用。
license: MIT
metadata:
  author: meta-workflow-generator
  version: "1.0"
---
```

## 正文模板

```markdown
# {{领域名称}}工作流编排器

协调 {{prefix}} 系列技能的端到端执行，管理进度、传递上下文、执行质量门禁。

**核心原则：** 编排器只做调度和决策，不做具体领域工作。每个环节的实际执行委派给对应技能。

## 进度追踪

使用进度表记录各技能节点的执行情况，编排器根据实际状况灵活判断下一步行动：

| # | 技能 | 完成情况 | 产出物 | 备注 |
|---|---|---|---|---|
| 1 | {{prefix}}1-{{name}} | ☐ 待执行 | — | — |
| 2 | {{prefix}}2-{{name}} | ☐ 待执行 | — | — |
| ... | | | | |

编排器根据依赖关系和实际情况判断下一个应执行的技能。无需严格状态机——Agent 应根据上下文灵活决策：
- 依赖已满足的节点可以执行
- 多个节点互不依赖时可并行（上限 3 个）
- 某节点暂时无法执行时，跳过并尝试其他可执行节点
- 用户表达意图变更时，灵活调整执行顺序

## 执行流程

### Phase 1：初始化

1. 读取工作流配置（技能列表、依赖关系、门禁配置）
2. 初始化进度追踪表
3. 识别无前置依赖的技能节点，确定起始点
4. 向用户展示流水线全景和预估步骤数

### Phase 2：执行循环

```
WHILE 存在未完成的技能节点:
    1. 判断哪些节点的依赖已满足，可以执行
    2. 选择下一个执行目标（优先级：依赖序号最小 > 关键路径优先）
    3. 准备上下文包（收集所有上游产出物）
    4. 调用对应技能执行
    5. 接收执行结果，执行质量门禁检查
    6. 更新进度表
    7. 向用户报告进度
```

**并行执行规则：** 当多个技能互不依赖且都可执行时，可并行执行。并行上限为 3 个（避免上下文过载）。

### Phase 3：质量门禁

每个技能完成后执行对应级别的质量检查：

| 门禁级别 | 适用条件 | 检查内容 | 未通过处理 |
|---|---|---|---|
| **严格** | 关键路径节点、多下游依赖 | 输出格式 + 内容质量 + 契约完整性 | 退回修正，最多 2 轮 |
| **标准** | 普通节点 | 输出格式 + 契约完整性 | 退回修正，最多 1 轮 |
| **宽松** | 辅助节点、无下游依赖 | 仅检查输出格式 | 标注问题，继续推进 |

**门禁配置表（生成时填充）：**

| 技能 | 门禁级别 | 原因 |
|---|---|---|
| {{prefix}}1-{{name}} | 严格 | 后续所有节点依赖其输出 |
| {{prefix}}2-{{name}} | 标准 | 有 1 个下游依赖 |
| ... | | |

### Phase 4：异常处理

| 异常场景 | 检测方式 | 处理策略 |
|---|---|---|
| **技能执行失败** | 技能报告错误或产出为空 | 分析失败原因 → 补充上下文后重试（最多 2 次）→ 仍失败则标记 `failed` 并询问用户 |
| **输出不符合契约** | 门禁检查不通过 | 明确缺失/错误项 → 退回对应技能修正 → 重新执行门禁 |
| **上游产出不完整** | 下游技能报告输入缺失 | 回溯上游技能 → 补充缺失部分 → 重新传递 |
| **用户要求跳过** | 用户明确表示跳过某节点 | 标记 `skipped` → 检查下游是否可降级运行 → 不可降级则标记下游为 `blocked` |
| **循环依赖检测** | 状态表中出现互相等待 | 立即中止 → 报告设计缺陷 → 建议用户调整流水线 |
| **超时/卡顿** | 单技能执行超过预期时间 3 倍 | 中断 → 询问用户是继续等待还是跳过 |

**降级策略：** 当某技能被跳过或失败时，下游技能可尝试以降级模式运行：
- 使用默认值/模板替代缺失输入
- 减少输出的详细度
- 标注 `[降级模式: 缺少 {上游技能} 的输出]`

### Phase 5：完成与交付

1. 所有技能 `completed` 或 `skipped` → 进入交付
2. 汇总所有产出物到最终交付目录
3. 生成执行报告：

```markdown
## 工作流执行报告

**场景：** {{任务描述}}
**耗时：** {{总轮次}} 轮对话

### 执行结果
| # | 技能 | 状态 | 门禁 | 重试次数 |
|---|---|---|---|---|

### 最终交付物
- {{产出物1路径}} — {{一句话描述}}
- {{产出物2路径}} — {{一句话描述}}

### 知识沉淀
{{执行中捕获的经验条目，如有}}

### 改进建议
{{对工作流本身的优化建议，如有}}
```

## 上下文注入规则

为每个技能准备的上下文包内容：

| 注入内容 | 来源 | 说明 |
|---|---|---|
| 上游产出物全文 | 对应技能的输出文件 | 直接提供全文，不要求技能自行读取 |
| 工作流全景摘要 | 编排器维护 | 让技能了解自己在流水线中的位置 |
| 当前进度状态 | 状态追踪表 | 已完成/待完成的节点列表 |
| 相关知识条目 | knowledge 目录 | 与当前技能相关的已有经验 |
| 用户偏好/修正 | 历史交互记录 | 用户在之前节点中表达的偏好 |

**注入原则：**
- 提供全文而非引用路径（技能应能独立工作）
- 只注入直接相关的上游产出（不注入全部历史）
- 上下文总量控制在 5000 字以内（超出时摘要化）

## 知识沉淀触发规则

执行过程中自动记录以下信号：

| 信号 | 记录内容 | 存储位置 |
|---|---|---|
| 技能重试 ≥ 2 次 | 失败原因 + 最终解法 | `{工作目录}/learnings-draft.md` |
| 门禁退回修正 | 缺陷模式 + 修正方式 | 同上 |
| 用户修正产出物 | 修正前后对比 + 修正原因 | 同上 |
| 降级模式触发 | 触发原因 + 实际影响 | 同上 |

## 进度报告格式

每完成一个技能后，向用户输出简短进度更新：

```
✓ {{技能名}} 已完成 ({{N}}/{{总数}})
  产出: {{产出物摘要，一句话}}
  下一步: {{下一个技能名}} — {{一句话说明}}
```

## 中断与恢复

用户可能在执行过程中中断。编排器通过进度表支持恢复：
- 每次技能完成后持久化进度表到 `{工作目录}/workflow-state.md`
- 恢复时读取进度表，从最后一个已完成节点之后继续
- 恢复时向用户确认："上次执行到 {{最后完成技能}}，是否从 {{下一个技能}} 继续？"
```

## 生成编排技能时的检查清单

| # | 检查项 |
|---|---|
| 1 | 状态机覆盖所有技能节点 |
| 2 | 依赖关系与步骤 2 的流水线设计一致 |
| 3 | 每个技能都有门禁级别配置 |
| 4 | 异常处理覆盖了所有 failed/blocked 路径 |
| 5 | 上下文注入规则为每个技能定义了具体内容 |
| 6 | 进度报告格式简洁且信息完整 |
| 7 | 中断恢复机制可用 |
