# 分支条件与异常处理设计模式

## 通用设计模式

| 模式 | 出现频率 | 说明 |
|---|---|---|
| 第一步必为"澄清/定义" | 全部 | 模糊输入必须先收敛为结构化框架 |
| 中间有"审查/验证"环节 | 全部 | 生成物需要独立评审才能进入下一步 |
| 最后一步是"整合/交付" | 全部 | 多个中间产物需要包装为最终交付物 |
| 核心理念 2-3 条 | 全部 | 精炼该环节的方法论精髓 |
| 质量自检清单 | 全部 | 每个技能都有明确的完成判断标准 |
| 可独立运行 | 全部 | 给定输入，每个技能不依赖会话历史 |

---

## 模式 1：输入质量分支

当上游产出物质量不一致时，下游技能需要根据输入质量选择不同执行路径。

**示例：数据分析工作流中的 analyst2-data-exploration**

```markdown
## 输入质量判断（步骤 1 执行前）

收到上游 analyst1 的分析框架后，先评估数据可用性：

| 输入状况 | 判断标准 | 执行路径 |
|---|---|---|
| 数据充足且质量高 | 缺失率 < 5%，无明显异常值 | → 正常路径：直接进入探索性分析 |
| 数据存在但质量差 | 缺失率 5-30%，或存在格式不一致 | → 清洗路径：先执行数据清洗子流程，再进入分析 |
| 关键数据缺失 | 核心字段缺失率 > 30%，或关键维度完全缺失 | → 阻塞路径：标记 blocked，回退到 analyst1 补充数据获取方案 |
| 数据源不可访问 | 连接失败、权限不足 | → 降级路径：基于已有样本数据做有限分析，标注 `[数据受限]` |

**分支写法规范：**
- 每个分支用 `→` 标记目标路径名称
- 阻塞路径必须说明回退到哪个上游技能
- 降级路径必须标注降级标记，让下游知道数据可靠性
```

## 模式 2：迭代循环与终止条件

当某环节需要反复打磨时，必须定义明确的终止条件防止无限循环。

**示例：内容营销工作流中的 draft ⇄ review 循环**

```markdown
## 审查-修正循环

### 循环逻辑

轮次 = 0
WHILE 轮次 < 3:
    轮次 += 1
    执行 content4-quality-review
    IF 所有维度通过 → 退出循环，进入 content5
    IF 存在阻塞性问题 → 退回 content3 修正对应部分
    IF 仅存在建议性问题 且 轮次 ≥ 2 → 接受当前版本，标注建议项，进入 content5

### 终止条件（满足任一即退出）

| # | 条件 | 说明 |
|---|---|---|
| 1 | 审查全维度通过 | 理想退出 |
| 2 | 迭代达 3 轮 | 强制退出，防止无限循环 |
| 3 | 连续两轮审查结果相同 | 收敛信号，继续迭代无增量价值 |
| 4 | 用户明确表示"够了" | 人为终止 |

### 问题分级

| 级别 | 定义 | 处理方式 |
|---|---|---|
| 阻塞性 | 不修正则无法发布（事实错误、逻辑矛盾、严重偏题） | 必须退回修正 |
| 建议性 | 修正后更好但不影响发布（措辞优化、结构微调） | 第 1 轮退回修正，第 2 轮起可接受 |
| 观察性 | 风格偏好、次要建议 | 记录但不退回 |
```

## 模式 3：条件分支（扇出选择）

当流水线中间需要根据条件选择不同的下游路径。

**示例：教育课程设计中的材料创建分支**

```markdown
## 教学材料类型路由

根据课程大纲中每个模块的学习目标层级（Bloom 分类），选择不同的材料创建策略：

### 路由规则

| 学习目标层级 | 材料策略 | 核心产出物 |
|---|---|---|
| 记忆/理解 | 讲解型 | 知识卡片 + 概念图 + 讲义 |
| 应用/分析 | 案例型 | 案例文档 + 练习题集 + 模板 |
| 评价/创造 | 项目型 | 项目说明书 + 评分标准 + 示范作品 |

### 混合模块处理

当单个模块同时包含多个层级的学习目标时：
1. 按目标拆分为子模块
2. 每个子模块独立路由到对应策略
3. 在汇总时按教学序列重新编排（低层级 → 高层级）
```

## 模式 4：异常恢复与降级

当技能执行失败时的降级和恢复策略。

**示例：数据分析工作流中 analyst3 执行失败的处理**

```markdown
## 异常处理策略

### 场景 1：统计方法不适用

**检测：** 数据不满足所选统计方法的前提假设（如正态性、独立性）
**处理：**
1. 记录原始方法和失败原因
2. 自动降级到非参数方法（如 t 检验 → Mann-Whitney U）
3. 在输出中标注 `[方法降级: {原方法} → {替代方法}, 原因: {前提假设不满足}]`
4. 继续执行，不回退

### 场景 2：样本量不足

**检测：** 效应量检验显示当前样本量无法达到统计显著性
**处理：**
1. 计算所需最小样本量
2. 报告给编排器：`BLOCKED: 需要 N 条数据，当前仅有 M 条`
3. 编排器决策：
   - 可获取更多数据 → 回退到 analyst2 扩展数据源
   - 无法获取 → 降级为描述性分析，标注 `[样本受限: 结论为趋势性观察，非统计验证]`

### 场景 3：外部数据源超时

**检测：** API 调用超时或数据库查询超过 60 秒
**处理：**
1. 重试 1 次（等待 30 秒）
2. 仍失败 → 检查是否有缓存/历史数据可用
3. 有缓存 → 使用缓存，标注 `[数据时效: 缓存于 {日期}]`
4. 无缓存 → 标记 `blocked`，报告给用户决策

### 降级标注规范

所有降级产出物必须在文档头部汇总降级信息：

> ## ⚠️ 降级说明
> - [方法降级] 假设检验使用非参数方法（原因：数据非正态分布）
> - [数据时效] 销售数据来自 2025-12-01 缓存（原因：数据源超时）
>
> 以上降级不影响主要结论的方向性判断，但置信度有所降低。
```

## 模式 5：用户干预点设计

在流水线中设置明确的用户决策点，平衡自动化与人工控制。

**示例：内容营销工作流中的干预点设计**

```markdown
## 用户干预点

### 必须干预（流程阻塞等待用户）

| 位置 | 干预内容 | 为什么不能自动化 |
|---|---|---|
| content1 完成后 | 确认选题方向 | 选题决定整个内容方向，错误代价极高 |
| content4 审查后 | 确认是否需要修正 | 涉及主观判断和品牌风格 |

### 可选干预（默认自动继续，用户可拦截）

| 位置 | 干预内容 | 自动行为 |
|---|---|---|
| content2 完成后 | 大纲确认 | 默认继续，用户无响应则自动推进 |
| content3 完成后 | 初稿预览 | 默认直接送审，用户可拦截自行修改 |

### 干预点在 SKILL.md 中的写法

在技能的输出规范中标注：

- 干预类型: 必须 | 可选
- 等待方式: 阻塞 | 超时自动继续（N秒）
- 用户可选操作:
  - 确认继续
  - 要求修正（附修正意见）
  - 跳过当前环节
  - 终止工作流
```

## 分支条件在 SKILL.md 中的写法规范

### 规范 1：条件判断表

用于输入质量分支或路由决策：

```markdown
## 输入路由

| 条件 | 判断方法 | 执行路径 | 降级说明 |
|---|---|---|---|
| {条件A} | {如何检测} | → 路径名称 | — |
| {条件B} | {如何检测} | → 路径名称 | {降级时的限制} |
| 以上均不满足 | — | → 默认路径 | — |
```

**规则：** 必须有默认路径（兜底），条件按优先级从高到低排列。

### 规范 2：循环结构

用于迭代审查/优化环节：

```markdown
## 迭代规则

- 最大轮次: {N}
- 退出条件（满足任一）:
  1. {理想退出条件}
  2. {强制退出条件}
  3. {收敛退出条件}
- 每轮产出: {中间产物描述}
- 超限处理: {达到最大轮次后怎么办}
```

### 规范 3：异常处理块

用于每个可能失败的步骤：

```markdown
## 异常处理

| 异常场景 | 检测方式 | 处理策略 | 是否回退 |
|---|---|---|---|
| {场景1} | {如何发现} | {做什么} | 是/否 → {回退到哪} |
| {场景2} | {如何发现} | {做什么} | 否，降级继续 |
```

**规则：** 每个"处理策略"必须是具体动作，不能是"适当处理"。回退时必须指明回退目标。
