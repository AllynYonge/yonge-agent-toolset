# 元技能工作流生成器

## 目标

根据用户的业务领域和任务需求，构建并生成一套**定制化技能工作流体系**。产出物是一组可直接安装使用的技能文件（SKILL.md + references），技能之间通过明确的输入/输出契约形成协作流水线。

## 核心约束

以下三条约束贯穿所有步骤，违反任何一条即为设计缺陷：

1. **解耦优先** — 每个生成的技能既是流水线一环，也是独立可用的专业工具。
   - 输入契约写"需要一份包含 A/B/C 的文档"，而非"读取技能 X 的输出"
   - 缺少上游产出时提供输入引导或降级运行，而非报错退出
   - description 描述独立使用场景，不写"在 X 之后使用"
   - 检验：把任意一个技能交给不知道其他技能的用户，他能独立完成任务

2. **不过度设计** — 3-8 个节点为宜，不为假想需求预留扩展。编排技能仅在节点 ≥ 6 且存在非线性依赖（分支/并行/循环）时生成。

3. **不写空壳** — 每个技能必须包含可操作的领域方法论。不确定时标注 `[基于通用方法论]`，不编造专业方法。

## 模式选择

| 条件 | 模式 | 说明 |
|---|---|---|
| 预估节点 ≤ 4，无复杂依赖 | 快速模式 | 3 步完成，1-2 轮对话 |
| 预估节点 ≥ 5，或有复杂依赖/并行/循环 | 完整模式 | 5 步完成，含架构确认 |
| 无法预估 | 先执行步骤 1，根据结果建议模式 | — |

用户可通过 `--quick` 或 `--full` 参数强制指定模式。

---

## 快速模式（≤ 4 节点）

### 步骤 1：领域理解 + 架构设计

一轮对话完成需求澄清和流水线设计。

**操作：**
1. 调用 `smart-search-cli` 搜索该领域的标准工作流、流程框架和行业 SOP
2. 向用户提问（≤ 3 个问题，优先选择题）明确领域、核心任务、质量标准
3. 基于用户回答 + 搜索结果设计流水线（环节识别 + 命名 + 依赖关系）
4. 以流水线总览表呈现，等待用户确认

**退出条件：** 用户确认流水线总览表。

### 步骤 2：逐技能生成

按流水线顺序，为每个技能生成完整文件。

**操作：**
1. 按 `references/skill-template.md` 生成 SKILL.md（含工作流协同章节 + 行为适配章节）
2. 生成 `references/domain-guide.md`，必须通过密度验证（≥1 专业框架、≥3 量化标准、≥2 反模式）
3. 生成 `references/output-template.md`（每个占位符附填写指引和示例值）
4. 如有工具依赖，声明工具集成层

**知识补充：** 全部技能生成后，统一询问用户一次："以上方法论有需要调整的部分吗？如有参考资料可提供。"用户不补充则保持现状。

**密度不达标时：** 对标注了 `[基于通用方法论]` 的技能，主动向用户追问结构化问题（决策规则/质量红线/反模式）以补充深度。

### 步骤 3：交付

1. 输出文件清单和使用说明
2. 执行完整性验证（质量门禁表）
3. 建议用户选一个技能做冒烟测试（可选，不阻塞交付）

## 完整模式（≥ 5 节点）

### 步骤 1：领域理解与需求澄清

通过对话理解用户的目标领域和任务特征。

**必须明确的信息：**

| 维度 | 问题 |
|---|---|
| 领域 | 这套工作流服务于什么领域？ |
| 核心任务 | 这个领域最典型的复杂任务是什么？ |
| 质量标准 | 什么算"做得好"？ |

**可选追问（能推断则不问）：** 现有痛点、使用者画像。

**提问策略：** 一次不超过 3 个问题，优先选择题。能从上下文推断的直接标注 `[假设]` 不追问。

**领域调研：** 用户明确领域后，调用 `smart-search-cli` 搜索该领域的标准工作流、流程框架和行业 SOP，作为后续环节拆分的参照基线。

**退出条件：** 能清晰描述核心任务的端到端流程、涉及的关键环节、以及每个环节的输入输出。

### 步骤 2：工作流架构设计

基于领域理解，设计技能流水线的整体架构。加载 `references/architecture-patterns.md` 辅助架构选择。调用 `smart-search-cli` 搜索同类领域成熟的流水线模式和最佳实践，作为架构决策的参照。

#### 2.1 环节识别

将核心任务分解为 **3-8 个** 有序环节。

**分解原则：**
- **单一职责** — 每个环节只做一件事
- **明确边界** — 环节之间通过文档传递，不共享隐式状态
- **可独立触发** — 给定输入即可工作
- **渐进交付** — 每完成一个环节就有可用的中间产物

**环节类型：**

| 类型 | 职责 |
|---|---|
| 输入澄清 | 从模糊需求收敛为结构化输入 |
| 策略/结构设计 | 制定方法论、选择路径、分解结构 |
| 内容生成 | 按规范产出核心交付物 |
| 质量审查 | 多维度评审与修正 |
| 整合交付 | 组合多个产出为最终交付物 |

#### 2.2 流水线设计

确定环节顺序、依赖关系和分支条件。

```
[技能1: 名称] —(产出物A)→ [技能2: 名称] —(产出物B)→ ...
                                          ↘ (条件X) → [技能2b: 分支]
```

#### 2.3 命名体系

**命名模式：** `{领域前缀}{序号}-{职责描述}`

示例：`content1-topic-research`、`analyst2-data-exploration`

#### 2.4 向用户确认架构

以流水线总览表呈现：

| 序号 | 技能名称 | 职责 | 输入 | 输出 | 依赖 |
|---|---|---|---|---|---|
| 1 | {name} | {一句话} | {来源} | {产出物} | 无 |

等待用户确认或调整后进入步骤 3。

### 步骤 3：逐技能详细设计与深度知识构建

按流水线顺序，为每个技能设计详细内容并构建方法论内核。每个技能的组成部分和模板见 `references/skill-template.md`。

**空壳技能不可接受** — 每个技能的 domain-guide 必须达到"领域从业者读了会觉得专业"的程度。

**每个技能需要设计：**

| 组成部分 | 内容 |
|---|---|
| 触发条件 | 什么情况下应该使用此技能 |
| 核心理念 | 此环节的方法论精髓（2-3 条） |
| 工作流程 | 分步骤的执行指导 |
| 输入契约 | 需要什么前置信息 |
| 输出规范 | 产出物的格式和质量标准 |
| 质量自检 | 交付前的检查清单 |
| 工作流协同 | 上下游衔接声明 |
| 行为适配 | 根据输入条件切换模式（≥1 个自适应维度） |
| 工具依赖 | 需要调用的外部工具及降级策略（如有） |

#### 三源知识汇聚

为每个技能节点按优先级逐源采集，直到方法论密度达标：

| 来源 | 操作 | 产出 |
|---|---|---|
| **源 1：行业标准检索** | 调用 `smart-search-cli` 技能搜索行业专业标准、SOP、最佳实践 | 权威方法论 + 行业规范 + 量化基准 |
| **源 2：结构化追问** | 用"专家访谈"式提问提取隐性知识 | 决策规则 + 边界案例 + 量化指标 |
| **源 3：模型领域知识** | 基于模型知识生成，标注 `[基于通用方法论]` | 方法论框架 + 行业通识 |

#### 结构化追问模板

不问泛泛的"你有什么经验"，而是问可产出规则的问题：

- **决策规则：** "这个环节，你根据什么条件选择不同的做法？"（→ 提取自适应分支）
- **硬规则：** "有没有'如果 X 则必须 Y'的强约束？"（→ 提取约束）
- **质量边界：** "做到什么程度算'够了'？有量化指标吗？"（→ 提取验证标准）
- **红线：** "什么样的产出物你会打回重做？"（→ 提取质量红线）
- **反模式：** "最常见的错误做法？看起来对但实际有害的做法？"（→ 提取 Never 规则）

#### 密度验证

每个技能的 domain-guide.md 必须通过以下检查：

| 检查项 | 不通过的信号 |
|---|---|
| 方法论框架 | 只有"分析→设计→执行"这种通用描述 |
| 判断标准 | 出现"适当"、"合理"、"酌情"等模糊词 |
| 专业术语 | 无该领域的专有名词或概念 |
| 量化指标 | 无任何数字、阈值、范围 |
| 反模式 | 无"不要做"的约束 |

不通过 → 追问用户或深入搜索补充，标注 `[待深化]`。

#### 自适应行为设计

每个技能应根据输入上下文自动调整行为。设计时必须回答：**这个技能在什么条件下走不同路径？**

| 自适应维度 | 示例 | 设计方式 |
|---|---|---|
| 输入复杂度 | 简单需求 vs 复杂需求 | 轻量模式 / 完整模式切换 |
| 领域/技术栈 | 前端 vs 后端 vs 全栈 | 路由到不同子流程 |
| 项目阶段 | 从零开始 vs 迭代已有 | 调整输入要求和输出深度 |
| 默认 | 无法判断时 | 走完整路径 |

**设计约束：**
- SKILL.md 正文 ≤ 3000 字，详细内容放 references/
- 使用祈使句写法
- 每个步骤有明确的验证方法
- ≥ 1 个自适应维度（不接受纯线性无分支的技能）

### 步骤 4：编排与文件生成

1. 定义上下文传递规则（文件路径约定、引用格式、缺失处理）
2. 若满足编排器条件（节点 ≥ 6 + 非线性依赖），按 `references/orchestrator-template.md` 生成编排技能
3. 在 `.claude/skills/` 下创建所有目录和文件
4. 为每个技能生成"工作流协同"章节（模板见 `references/skill-template.md` 末尾）

若技能含分支/循环/异常处理，参考 `references/branching-patterns.md` 设计条件逻辑。

### 步骤 5：交付与可选验证

#### 5.1 完整性验证

| 检查项 | 标准 |
|---|---|
| 流水线覆盖 | 核心任务的每个环节都有对应技能 |
| 输入输出闭合 | 每个技能的输入都能从上游获得或独立获取 |
| 无孤立节点 | 每个技能至少被一个其他技能引用或产出最终交付物 |
| 命名一致 | 前缀、编号、术语统一 |
| 可独立触发 | 给定输入，每个技能可独立运行 |

#### 5.2 交付

输出文件清单和使用说明。建议验证方案（详见 `references/validation-guide.md`），用户可选择执行或跳过。

## 质量门禁

生成的每个技能必须满足：

| 维度 | 标准 |
|---|---|
| 自包含 | 不依赖会话历史，给定输入即可工作 |
| 可验证 | 每个步骤有明确的完成标志 |
| 适度详细 | SKILL.md ≤ 3000 字，详细内容在 references/ |
| 风格一致 | 祈使句、无废话、结构统一 |
| 领域专业 | 包含该领域的专业方法论，非泛泛之谈 |
| domain-guide 密度 | 必须包含：≥1 个专业框架/方法论名称、≥3 条量化判断标准、≥2 条反模式 |
| output-template 精度 | 模板中的每个占位符附带填写指引和示例值 |
| 自适应覆盖 | ≥1 个维度的行为切换（不接受纯线性无分支的技能） |

## 约束规则

1. **领域聚焦** — 围绕用户指定的任务范围，不限于编码或非编码
2. **不过度设计** — 3-8 个节点为宜，编排技能仅在节点 ≥ 6 且有非线性依赖时生成
3. **不重复造轮子** — 如果现有技能可直接复用，引用而非重建
4. **不写空壳技能** — 每个技能必须包含可操作的领域方法论
5. **假设标注** — 缺乏领域知识时标注 `[假设]` 或 `[基于通用方法论]`，不编造
6. **解耦优先** — 每个技能独立可用，缺少上游时降级而非阻塞。this is NON-NEGOTIABLE
