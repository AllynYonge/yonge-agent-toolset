# skill-merge

将一个或多个**源技能**的能力低破坏性地整合到**目标技能**中。

核心约束：**目标技能的现有行为在合并后必须完全不变**。合并只做增量扩展，不做替换。

---

## 触发场景

| 情境 | 说明 |
|---|---|
| 两个技能功能重叠 | 用户发现两个技能做类似的事，想合二为一 |
| 能力迁移 | 用户想把 A 技能的某些能力搬到 B 技能中 |
| 技能整合 | 多个小技能合并为一个综合技能 |
| 数据层增强 | 把一个技能的数据资产（CSV、规则、模板）注入另一个技能 |

---

## 工作流阶段

### Phase 1：分析（Analysis）

**目标：** 理解所有参与技能的能力边界、重叠区域、和架构差异。

支持 1:1（一个源 → 一个目标）和 N:1（多个源 → 一个目标）两种模式。
多源模式下，对每个源技能独立执行分析，最后汇总重叠和冲突。

**步骤：**

1. **读取目标技能和所有源技能的 SKILL.md**，逐个提取：
   - 触发条件（何时使用）
   - 工作流程（做什么）
   - 输出产物（产出什么）
   - 设计哲学（为什么这样做）
   - 技术栈（脚本语言、依赖、运行时）

2. **扫描文件结构**，对比：
   - 脚本目录（scripts/）
   - 参考文档（reference/ 或 references/）
   - 数据资产（data/、templates/）
   - 配置文件

3. **生成对比矩阵**，输出：

```markdown
## 对比分析

| 维度 | 目标技能 | 源技能 A | 源技能 B | ... |
|------|---------|---------|---------|-----|
| 定位 | ... | ... | ... | ... |
| 技术栈 | ... | ... | ... | ... |
| 数据来源 | ... | ... | ... | ... |
| 输出格式 | ... | ... | ... | ... |

## 能力重叠

| 重叠领域 | 目标技能做法 | 源技能(s)做法 | 冲突程度 | 涉及源 |
|---------|------------|-------------|---------|--------|
| ... | ... | ... | 低/中/高 | A / B / ... |

## 迁移价值评估

| 能力 | 来源 | 迁移价值 | 理由 |
|------|------|---------|------|
| ... | 源技能 A | 高/中/低/不迁移 | ... |
| ... | 源技能 B | 高/中/低/不迁移 | ... |

**多源冲突检测：** 当多个源技能提供同一能力的不同实现时，标记为"源间冲突"，
在 Phase 2 中决定采用哪个版本或合并两者。
```

4. **与用户确认**迁移范围——哪些能力迁移、哪些不迁移、优先级如何。
   多源模式下，额外确认：源间冲突的处理策略、是否分批迁移（推荐）或一次全部完成。

---

### Phase 2：架构设计（Architecture）

**目标：** 设计最小侵入的集成方案。

**核心原则（按优先级排序）：**

1. **不修改目标技能现有文件**（SKILL.md 除外，且仅追加）
2. **不修改源技能的任何文件**
3. **技术栈统一**——新增脚本匹配目标技能的语言和风格
4. **渐进启用**——扩展目录不存在时行为完全不变
5. **可独立回滚**——删除扩展目录即完全恢复原状
6. **冲突规则明确**——目标技能原有规则 > 扩展推荐

**标准集成架构：插件层（Plugin Layer）**

```
target-skill/
├── SKILL.md                    ← 末尾追加 Extensions 段（唯一改动）
├── reference/                  ← 不动
├── scripts/                    ← 不动
│
└── extensions/                 ← 新增：全部迁移内容在此
    ├── EXTENSIONS.md           ← 扩展注册表（能力清单 + 使用说明）
    ├── data/                   ← 迁移的数据资产
    ├── rules/                  ← 迁移的规则文档
    ├── templates/              ← 迁移的模板文件
    └── scripts/                ← 新增脚本（匹配目标技能风格）
```

**架构决策清单（逐项确认）：**

- [ ] 信息层级：源技能的产物在目标技能的信息架构中处于什么位置？
- [ ] 数据流向：新增产物是权威源还是派生物？谁消费它？
- [ ] 冲突规则：两套规则矛盾时谁优先？
- [ ] 脚本接口：CLI 参数、输入格式、输出格式是什么？
- [ ] 与现有命令的协作：哪些现有命令会调用扩展能力？怎么调用？

**输出：** 架构设计文档，包含目录结构、脚本规格、集成点、冲突规则。

---

### Phase 3：执行（Implementation）

**分阶段执行，每阶段独立可验证：**

#### Stage A：数据层迁移

- 创建 `extensions/` 目录结构
- 复制数据文件（CSV、JSON、YAML 等）
- 创建规则文档（翻译/格式转换如需要）
- 创建模板文件
- 创建 `EXTENSIONS.md` 注册表

**多源模式注意事项：**
- 来自不同源技能的数据文件放入同一个 `extensions/data/` 目录，用子目录或前缀区分来源
- 如果多个源技能有同名文件（如都有 `colors.csv`），合并为一个文件或用来源前缀区分
- EXTENSIONS.md 中按来源分组记录能力清单

**验证：** 文件存在 + 行数一致 + 目标技能现有命令行为不变

#### Stage B：脚本实现

- 按目标技能的代码风格编写新脚本
- 匹配现有脚本的：
  - 导入风格（ESM/CJS/Python import）
  - 错误处理模式
  - 输出格式（JSON to stdout / 文件写入）
  - CLI 参数解析方式
- 不引入目标技能未使用的外部依赖

**验证：** 每个脚本独立运行成功 + 输出格式正确

#### Stage C：SKILL.md 集成

- 在 SKILL.md **末尾**追加 Extensions 段落
- 段落结构：
  ```markdown
  ## Extensions (Optional ...)

  When `extensions/` directory exists, the following capabilities activate.
  All existing commands, references, and [laws/rules] remain authoritative.
  **Conflict resolution: existing [laws/rules] > extension recommendations.**

  ### [能力 1 名称]
  [一句话描述 + 使用命令]

  ### [能力 2 名称]
  ...
  ```
- 不修改 SKILL.md 的任何现有内容

**验证：** `diff` 确认只有追加

#### Stage D：端到端验证

执行验证矩阵：

| 场景 | 操作 | 预期 |
|------|------|------|
| 无 extensions/ | 正常使用目标技能 | 行为与合并前完全一致 |
| 有 extensions/ | 使用新增能力 | 正常工作 |
| 有 extensions/ | 使用原有命令 | 行为不变 |
| 删除 extensions/ | 重新使用所有命令 | 完全恢复，无报错 |

---

### Phase 4：文档与收尾（Documentation）

**输出变更摘要：**

```markdown
## 合并完成

**更改的文件：** [列出]
**新增的文件：** [列出，按类别分组]
**故意没碰的文件：** [列出]
**需要跟进的内容：** [列出]
```

## 决策框架

### 什么应该迁移

| 条件 | 迁移 |
|------|------|
| 源技能独有 + 对目标技能有明确增益 | 是 |
| 源技能的数据资产（CSV、知识库） | 是（复制，不移动） |
| 源技能的规则/优先级体系 | 是（翻译/格式化后） |
| 源技能的模板/规格文档 | 是 |

### 什么不应该迁移

| 条件 | 原因 |
|------|------|
| 目标技能已有等价能力 | 冗余，增加维护成本 |
| 与目标技能技术栈不兼容的脚本 | 需要重写而非复制 |
| AI 模型训练数据中的高频知识（如 Tailwind 语法） | 参考文档增益极低 |
| 源技能的内部实现细节（__pycache__、test/） | 不是能力，是实现 |
| 与目标技能设计哲学冲突的规则 | 会造成内部矛盾 |

### 冲突处理策略

当源技能和目标技能对同一问题有不同答案时：

1. **目标技能有明确规则** → 保留目标技能的规则，不迁移冲突内容
2. **目标技能无规则，源技能有** → 迁移为"建议"而非"规则"
3. **两者规则互补不矛盾** → 合并，标注来源
4. **两者规则直接矛盾** → 暂停，向用户呈现两种选择，等待决策

---

## 质量标准

合并完成后必须满足：

- [ ] 目标技能的所有现有命令行为不变
- [ ] 源技能未被修改
- [ ] 新增脚本可独立运行且输出格式正确
- [ ] `extensions/` 目录删除后目标技能完全恢复
- [ ] SKILL.md 的修改仅为末尾追加
- [ ] 无新增外部依赖（或已与用户确认）
- [ ] 冲突规则已明确声明
- [ ] EXTENSIONS.md 注册表完整记录了所有新增能力
