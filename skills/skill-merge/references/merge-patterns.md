# 合并架构模式

对比四种常见的技能整合架构，帮助在 Phase 1 快速选型。

---

## 分析输出模板

Phase 1 第 3 步的对比矩阵使用以下格式：

```markdown
## 对比分析

| 维度 | 目标技能 | 参考技能 A | 参考技能 B | ... |
|------|---------|---------|---------|-----|
| 定位 | ... | ... | ... | ... |
| 技术栈 | ... | ... | ... | ... |
| 数据来源 | ... | ... | ... | ... |
| 输出格式 | ... | ... | ... | ... |

## 能力重叠

| 重叠领域 | 目标技能做法 | 参考技能(s)做法 | 冲突程度 | 涉及源 |
|---------|------------|-------------|---------|--------|
| ... | ... | ... | 低/中/高 | A / B / ... |

## 迁移价值评估

| 能力 | 来源 | 迁移价值 | 理由 |
|------|------|---------|------|
| ... | 参考技能 A | 高/中/低/不迁移 | ... |
| ... | 参考技能 B | 高/中/低/不迁移 | ... |

**多源冲突检测：** 当多个参考技能提供同一能力的不同实现时，标记为"源间冲突"，
在 Phase 2 中决定采用哪个版本或合并两者。
```

## 模式 A：插件层（Plugin Layer）

**结构（单源 1:1）：**
```
target-skill/
├── [existing files untouched]
└── extensions/
    ├── EXTENSIONS.md
    ├── data/
    ├── rules/
    ├── templates/
    └── scripts/
```

**结构（多源 N:1）：**
```
target-skill/
├── [existing files untouched]
└── extensions/
    ├── EXTENSIONS.md              ← 统一注册表，按来源分组记录
    ├── source-skill-a/            ← 每个源技能一个子目录
    │   ├── data/
    │   ├── rules/
    │   ├── templates/
    │   └── scripts/
    ├── source-skill-b/
    │   ├── data/
    │   └── scripts/
    └── shared/                    ← 可选：跨源共享的脚本/工具
        └── scripts/
```

**多源目录规则：**
- 每个源技能的迁移内容放入以源技能名命名的子目录
- `EXTENSIONS.md` 留在 `extensions/` 根目录，按来源分节记录
- 多个源共用的脚本（如通用搜索引擎）放入 `shared/scripts/`
- 回滚某个源 = 删除对应子目录 + 更新 EXTENSIONS.md

**单源时是否也用子目录？**
不强制。单源时平铺更简洁（路径短），符合 YAGNI 原则。
如果预期未来会有更多源合入，可以一开始就用子目录结构。

**适用场景：**
- 源技能有独立的数据资产（CSV、知识库）
- 源技能的能力可以表达为"查询"或"生成"操作
- 目标技能有明确的脚本调用模式

**优点：**
- 零侵入（只追加 SKILL.md）
- 完全可逆（删目录即恢复）
- 职责清晰（extensions/ = 全部新增内容）
- 多源时来源可追溯，可独立回滚单个源

**缺点：**
- 目录层级深（多源时 `extensions/source-a/scripts/search-data.mjs`）
- 需要在 SKILL.md 中声明激活条件
- 多源时 EXTENSIONS.md 维护成本略高

**实际案例：** Impeccable + UI-UX-Design 合并（单源，平铺结构）

## 模式 B：Sidecar 文件

**结构：**
```
target-skill/
├── SKILL.md
├── reference/
│   ├── [existing refs]
│   └── ext-*.md          ← 新增参考文档，用前缀区分
└── scripts/
    ├── [existing scripts]
    └── ext-*.mjs         ← 新增脚本，用前缀区分
```

**适用场景：**
- 源技能的能力主要是"知识型"（参考文档、规则清单）
- 不需要独立的数据层或搜索引擎
- 目标技能的 reference/ 加载机制已经成熟

**优点：**
- 与现有结构融合更紧密
- 无需额外的 EXTENSIONS.md 注册表
- 目标技能的命令可以自然地加载 `ext-*` 参考

**缺点：**
- 回滚需要逐个删除 `ext-*` 文件（不如删整个目录方便）
- 容易与现有文件混淆
- 不适合大量数据文件

**适用场景举例：** 将一套设计规则整合到已有技能的参考目录中

---

## 模式 C：适配器脚本（Adapter）

**结构：**
```
target-skill/
├── SKILL.md
├── scripts/
│   ├── [existing scripts]
│   └── adapter.mjs       ← 单个适配器，封装对源技能的调用
└── [source skill remains separate, adapter calls it]
```

**适用场景：**
- 源技能保持独立运行（不复制数据）
- 两个技能需要协作但不合并
- 源技能更新频繁，不想维护副本

**优点：**
- 源技能数据不重复存储
- 源技能可独立更新，目标技能自动受益
- 最小的文件增量

**缺点：**
- 运行时依赖源技能存在
- 源技能被删除则适配器失效
- 两个技能的耦合度高于模式 A

**适用场景举例：** 目标技能在特定步骤调用源技能的搜索脚本

## 模式 D：联动声明（Synergy Declaration）

**结构：**
```
target-skill/
├── SKILL.md          ← 末尾追加「## 联动（Synergy）」章节（唯一改动）
├── reference/        ← 不动
├── scripts/          ← 不动
└── [no new files]
```

**适用场景：**
- 两个技能各自独立，不需要合并能力或迁移数据
- 在特定工作流中存在自然的协作触发点（一方输出 = 另一方输入）
- 希望在目标技能中留下协作指引，帮助 Agent/用户发现联动路径
- 参考技能更新频繁，不想产生任何耦合

**优点：**
- 零文件增量（不创建目录、不复制文件）
- 完全可逆（删段落即恢复）
- 参考技能零变更、零依赖
- 目标技能独立工作能力不受影响
- 维护成本极低

**缺点：**
- 不提供任何运行时能力增强（纯文档层面）
- 联动依赖用户/Agent 主动触发，无自动化
- 协作场景变化时需手动更新章节

**追加章节模板：**
```markdown
---

## 联动（Synergy）

以下技能与本技能存在协作关系。各技能保持独立，联动通过工作流衔接实现。

### × [参考技能名]

| 场景 | 本技能职责 | [参考技能名] 职责 | 衔接方式 |
|------|-----------|-----------------|---------|
| [场景描述] | [本技能做什么] | [协作技能做什么] | [如何触发/传递] |

**协作原则：**
- 本技能不依赖 [参考技能名] 即可独立工作
- 联动时本技能产出物即为 [参考技能名] 的输入
- 冲突时以本技能的设计决策为准
```

多个参考技能时，每个参考技能一个 `### ×` 子章节。

**执行步骤（Phase 2–3 聚焦于联动设计）：**

1. **Phase 2（架构设计）：** 读取双方 SKILL.md，提取触发条件、工作流阶段、输出产物、设计哲学；识别协作触发点（一方输出是否为另一方自然输入、顺序/并行场景、互补覆盖）；设计联动章节结构
2. **Phase 3（执行）：** 生成联动场景表与用户确认，在目标技能 SKILL.md 末尾追加联动章节（使用上方模板）
3. **验证：** `diff` 确认仅末尾追加 + 参考技能零变更

**变更摘要格式：**

```markdown
## 联动声明完成

**修改的文件：** [目标技能]/SKILL.md（末尾追加）
**新增的文件：** 无
**参考技能变更：** 无
**联动场景数：** [N] 个
```

多个参考技能时，每个参考技能一个 `### ×` 子章节。

## 选型决策树

```
需要迁移能力/数据，还是只声明协作关系？
├─ 只声明协作关系 → 模式 D（Synergy Declaration）
│   不迁移文件，仅追加联动章节
└─ 需要迁移能力/数据
    ├─ 多个源技能？
    │   ├─ 是 → 模式 A（Plugin Layer，多源子目录结构）
    │   │       每个源一个子目录，shared/ 放共用脚本
    │   └─ 否（单源）
    │       ├─ 源技能有大量数据资产（CSV/JSON/知识库）？
    │       │   ├─ 是 → 模式 A（Plugin Layer，平铺结构）
    │       │   └─ 否
    │       │       ├─ 源技能需要保持独立更新？
    │       │       │   ├─ 是 → 模式 C（Adapter）
    │       │       │   └─ 否
    │       │       │       ├─ 迁移内容主要是文档/规则？
    │       │       │       │   ├─ 是 → 模式 B（Sidecar）
    │       │       │       │   └─ 否 → 模式 A（Plugin Layer，平铺结构）
    │       │       │       └─
    │       │       └─
    │       └─
    └─
```

## 混合使用

实际项目中可能混合多种模式：

- **数据密集型能力** → Plugin Layer（extensions/data/）
- **知识型能力** → 写入 extensions/ 的 rules/ 或 templates/
- **需要保持同步的能力** → Adapter 调用源技能
- **独立但有协作点的技能** → Synergy Declaration（纯文档声明）

关键是保持一致性：同一次合并中只用一种主模式，避免新增内容散落在多个不相关的位置。模式 D 可以与 A/B/C 叠加——先用 A/B/C 迁移能力，再用 D 声明与其他技能的联动关系。
